# sblg-spot — 기회 포착

이미 돈이 흐르고 있지만 만족이 없는 지점을 찾는 스킬. 아이디어를 만들지 않는다. 해결책을 제안하지 않는다. 고통과 지출만 기록한다. 처방 전에 진단.

진단의 원칙은 간단하다: 가장 먼 거리에서 가장 가까운 거리로 접근한다. 먼저 관찰하고, 시장의 무의식적 신호를 감지하고, 구조화된 질문을 던지고, 마지막에 직접 손을 댄다. 최종 판정은 세 가지 중 하나.

## Commands

| 커맨드 | 설명 |
|--------|------|
| `/sblg-spot` | 새 기회 탐색 시작 (Phase 1부터) |
| `status` | 현재 진행 상태 |
| `continue` | 이전 대화에서 이어서 (\_status.md 기반) |

## Session Continuity

새 대화에서 `continue` 또는 `status` 입력 시:
1. `~/.sblg/spot/` 디렉토리에서 가장 최근 세션 탐색
2. `_status.md` 읽어서 현재 Phase 파악
3. 남은 작업부터 이어서 진행

## Web Search

검색이 필요한 단계에서 사용 가능한 웹 검색 도구를 사용한다.
검색 도구가 없으면 "검색 불가 — 내부 지식으로 진행" 안내 후 계속.

## 절대 하지 않는 것

- 해결책 제안 ("이렇게 만들면 됩니다" 금지 — 처방은 sblg-solve의 몫)
- 새로운 시장이나 카테고리 제안 (이미 돈이 흐르는 곳만 본다)
- 기존 솔루션 이름과 가격이 특정되지 않는 항목 포함
- 검증 불가능한 시장 규모 추정

## Workflow

### Phase 1: 관찰 — 시장을 말없이 본다

환자를 만나면 말하기 전에 먼저 본다. 안색, 자세, 기색. 시장도 마찬가지다. 이 단계에서는 판단하지 않는다. 보기만 한다.

**AskUserQuestion으로 도메인 입력을 받는다:**

"탐색하고 싶은 도메인이나 분야를 알려주세요. 그리고 그 도메인에 관심을 갖게 된 맥락이 있으면 함께 알려주세요. (예: '프리랜서 디자이너의 프로젝트 관리 — 내가 프리랜서라서 직접 겪고 있다')"

사용자가 도메인을 입력하면:

**Step 1: 기존 솔루션 지도 작성**

도메인에 이미 존재하는 솔루션들을 나열:
- 솔루션 이름
- 가격 구조 (무료/프리미엄/유료, 월 얼마)
- 랜딩 페이지가 약속하는 핵심 가치
- 타겟 고객층

**Step 2: 돈의 흐름 관찰**

어디에 얼마가 흐르고 있는지 파악:
- 가장 비싼 솔루션은 무엇이고 왜 비싼가
- 가장 많은 사용자를 가진 솔루션은 무엇인가
- 최근 투자를 받은 솔루션이 있는가

이 단계에서 WebSearch로 도메인의 주요 솔루션 정보를 수집한다.

완료 후 보고: "Phase 1 완료: 도메인 내 솔루션 N개 확인. 월 지출 범위: $X ~ $Y."

산출물: `observation.md`

### Phase 2: 감지 — 시장이 무의식적으로 내는 신호를 잡는다

사용자가 의식적으로 표현하지 않아도 행동으로 드러나는 것들을 잡는다. 목소리 톤, 호흡 소리를 감지하듯.

**WebSearch로 탐색할 것:**
- 커뮤니티 불만 게시글 (Reddit, 블라인드, 디스콰이엇, 트위터, 앱 리뷰)
- "대안 찾습니다", "이거 대체할 만한 거 없나요" 류의 글
- 리뷰 사이트 별점과 부정적 댓글 패턴
- "해지했습니다", "다시 수동으로 돌아갔습니다" 류의 이탈 신호

**증거 위계:**

전해 들은 말을 믿지 말고, 직접 확인한 것만 기록한다:
- 직접 관찰 (리뷰 원문, 게시글 원문) → 가장 신뢰
- 교차 검증 (2개 이상 독립 출처에서 같은 불만) → 신뢰
- 단일 출처 → 참고만
- 업계 상식, 전해 들은 말 → 증거로 인정하지 않음

완료 후 보고: "Phase 2 완료: N건의 불만 신호 수집. 교차 검증된 패턴 M개."

산출물: `signals.md`

### Phase 3: 전문가 패널 질문 — 구조화된 심문

구조화된 질문으로 도메인을 깊이 파고든다. 해당 도메인의 Top 0.1% 실무 전문가 페르소나 3~5명을 자동 생성한다.

**페르소나 생성 기준:**
- 같은 분야 2명 (실무에서 직접 돈을 쓰는 당사자 관점)
- 인접 분야 1~2명 (유사한 지출 패턴을 다른 산업에서 본 사람)
- 각 페르소나에 이름, 전문 영역, 경력 배경 부여

**병렬 subagent로 각 전문가가 독립적으로 탐색:**
1. "이 도메인에서 가장 비싼데 가장 불만인 솔루션은?"
2. "매일 반복되는데 기존 솔루션이 해결 못 하는 고통은?"
3. "돈을 쓰면서도 '차라리 수동으로 하겠다'고 느끼는 것은?"
4. "경쟁 솔루션이 많은데도 아무도 만족 못 하는 영역은?"

Phase 1~2에서 수집한 관찰과 신호를 전문가에게 입력으로 제공한다. 전문가는 이미 발견된 신호를 확인하거나 반박하고, 놓친 지점을 추가한다.

완료 후 보고: "Phase 3 완료: 전문가 N명 의견 수집. 누수 지점 후보 M개."

산출물: `expert-inquiry.md`

### Phase 4: 촉진과 분해 — 직접 증거에 손을 댄다

맥을 짚듯, 후보 누수 지점에 직접 손을 댄다. Phase 1~3에서 나온 모든 후보를 교차 검증하고, "왜 돈이 여기 흐르는가"를 네 가지 관점에서 분해한다.

**3가지 조건을 모두 충족하는 것만 누수 지점으로 확정:**
1. 기존 지출 존재 — 이 문제를 해결하려고 이미 특정 솔루션에 돈을 쓰고 있다 (솔루션 이름과 가격 명시)
2. 반복 고통 — 일회성이 아닌 매일/매주/매월 반복된다
3. 기존 솔루션 불만 — 돈을 쓰면서도 불만인 구체적 이유가 있다

**증거 기준: 2개 이상 독립 출처에서 교차 확인된 것만 확정. 단일 출처는 탈락.**

**각 확정된 누수 지점에 대해 네 가지 관점에서 분해:**

무언가가 왜 현재 형태로 존재하는지 완전히 이해하려면 네 가지를 모두 답해야 한다:

- **돈의 구조**: 이 솔루션에 실제로 흘러가는 돈의 규모와 구조는? (월 구독, 건당 과금, 인건비 등)
- **형태**: 이 솔루션이 취하고 있는 형태는? (SaaS, 에이전시, 수동 노동, 엑셀 등)
- **원인**: 무엇이 사람들을 이 솔루션에 밀어넣었나? (대안 부재, 관행, 전환 비용, 상사의 지시 등)
- **목적**: 사용자가 진짜 원하는 것은? 이 솔루션이 실제로 달성하는 것과의 간극은?

**목적과 형태의 불일치가 곧 누수 지점의 크기다.** 사용자가 원하는 것과 솔루션이 실제로 제공하는 것의 간극이 클수록 기회가 크다.

최소 5개, 최대 7개 확정.

**각 누수 지점 출력 형식:**

```
### 누수 지점 [번호]: [고통 한 문장]
- 누가 겪는가: (직업, 상황, 조직 규모)
- 현재 쓰는 솔루션: (이름)
- 월 지출: $__ / ₩__
- 왜 구린가: (구체적 불만 3가지)
- 빈도: 매일/매주/매월
- 증거: (실제 불만 출처 — URL, 커뮤니티명, 리뷰 사이트)
- 목적-형태 간극: (사용자가 원하는 것 vs 실제 제공하는 것)
```

완료 후 보고: "Phase 4 완료: 검증된 누수 지점 N개 확정. 탈락 M개 (증거 부족)."

산출물: `verified-leaks.md`

### Phase 5: 판정 — 진단서 작성

**각 누수 지점에 세 가지 판정 중 하나:**

1. **다룰 수 있는 문제** — 기회 명확. 지출 크고, 빈도 잦고, 불만 심하고, 목적-형태 간극이 크다.
2. **싸워볼 만한 문제** — 불확실하지만 조사할 가치 있다. 일부 조건은 강하나 전부 강하지는 않다.
3. **손대지 않을 문제** — 고통은 실재하나 구조적으로 해결 불가능하거나 시장이 너무 작다.

**기회 밀도 스코어링:**
- 월 지출 규모 (1~5)
- 고통 빈도 (1~5)
- 불만 강도 (1~5)
- 목적-형태 간극 (1~5)
- 기회 스코어 = 네 항목의 곱 (/625)

"다룰 수 있는 문제" 판정을 받은 것 중 스코어 순으로 TOP 3 선정. 각각 한 줄로 선정 이유.

완료 후 보고: "Phase 5 완료: 다룰 수 있는 문제 N개, 싸워볼 만한 문제 M개, 손대지 않을 문제 K개. TOP 3 선정."

산출물: `verdict.md`

## Output Structure

```
~/.sblg/spot/
└── {도메인슬러그}-{YYYYMMDD}/
    ├── _status.md
    ├── observation.md
    ├── signals.md
    ├── expert-inquiry.md
    ├── verified-leaks.md
    └── verdict.md
```

## _status.md Format

```markdown
# Session Status
- domain: {도메인}
- current_phase: {1~5}
- started_at: {날짜}
- last_updated: {날짜}
- solutions_found: {N}
- signals_collected: {N}
- expert_count: {N}
- leaks_verified: {N}
- leaks_dropped: {N}
- top3: {미정 또는 목록}
- next_action: {다음에 해야 할 구체적 작업}
```
