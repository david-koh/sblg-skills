# sblg-build — 첫 번째 매출 설계

검증된 아이디어를 14일 안에 첫 번째 유료 전환으로 바꾸는 실행 설계서를 만드는 스킬. 시장을 분석하지 않는다. 아이디어를 평가하지 않는다. 완성도 높은 제품을 설계하려 하지 않는다. 지금 당장 무엇을, 어떻게, 누구에게, 얼마에 팔지만 결정한다.

sblg-spot이 기회를 발굴하고, sblg-solve가 해결책을 도출했다면, 이 스킬은 그 다음 질문에 답한다: **어떻게 만들고, 어떻게 팔 것인가.**

제약 속에서 결정적 행동을 내는 순서는 정해져 있다: 기반을 다지고, 지형을 읽고, 말을 준비하고, 싸우고, 판단을 끝낸다.

## Commands

| 커맨드 | 설명 |
|--------|------|
| `/sblg-build` | 새 빌드 플랜 시작 |
| `status` | 현재 진행 상태 |
| `continue` | 이전 대화에서 이어서 (\_status.md 기반) |

## Session Continuity

새 대화에서 `continue` 또는 `status` 입력 시:
1. `~/.sblg/build/` 디렉토리에서 가장 최근 세션 탐색
2. `_status.md` 읽어서 현재 Phase와 진도 파악
3. 남은 작업부터 이어서 진행

## Web Search

검색이 필요한 단계에서 사용 가능한 웹 검색 도구를 사용한다.
검색 도구가 없으면 "검색 불가 — 내부 지식으로 진행" 안내 후 계속.

## 절대 하지 않는 것

- 아이디어 단계에서 시작하지 않는다 — 검증 근거 없이 진행하면 sblg-spot을 먼저 권고한다
- 핵심 기능을 2개 이상 허용하지 않는다 — 두 번째 기능을 쓰면 다시 하나를 고르도록 돌려보낸다
- 시장 규모를 추정하지 않는다
- 완성도를 이유로 배포를 늦추지 않는다

## Workflow

### Phase 0: 입력 확인

sblg-solve의 산출물이 있는지 확인한다.

- `~/.sblg/solve/` 에서 가장 최근 세션의 `prototype.md`가 있으면 → 읽고 Phase 1로 진행. 사용자에게 "sblg-solve의 최소 증명을 기반으로 진행합니다. 맞나요?" 확인한다 (AskUserQuestion).
- `prototype.md` 없이 `solutions.md`만 있으면 → "sblg-solve에 최소 증명(Phase 5)이 아직 없습니다. sblg-solve를 먼저 완료하시겠어요, 아니면 여기서 직접 제품 개념을 잡을까요?" (AskUserQuestion)
- sblg-solve 산출물 자체가 없으면 → **AskUserQuestion으로 직접 입력받는다:**

"어떤 제품/서비스를 만들려고 하시나요? 아래를 알려주세요:
1. 무엇을 만들 것인가 (한 줄)
2. 누구를 위한 것인가
3. 이 아이디어가 검증됐다는 근거 (실제 대화, 선결제 의사, 기존 유사 솔루션의 유료 사례 중 하나)"

검증 근거가 없으면 진행을 멈추고 sblg-spot을 먼저 권고한다.

---

### Phase 1: 압축 — 다 버리고 하나를 남긴다

적으면 얻고, 많으면 길을 잃는다. 14일과 솔로 빌더라는 제약에서 기능 하나를 더 추가하는 것은 경쟁력이 아니라 방향 상실이다. 이 단계의 목적은 아이디어에서 더 이상 뺄 것이 없는 상태까지 압축하는 것이다.

**Step 1: 제약 확인 (AskUserQuestion)**

시작 전 세 가지를 확정한다:

"아래 제약을 확인해주세요:
- 개발 인원: 솔로인가요, 팀인가요?
- 타임라인: 14일이 맞나요? (단축/연장 이유가 있으면 알려주세요)
- 초기 운영 예산: 첫 달 기준 얼마인가요?"

**Step 2: 제품 압축**

세 가지를 확정한다:
- **이름**: 설명하지 않아도 짐작되는 이름. 설명이 필요하면 다시
- **한 문장 설명**: "누가, 무엇을, 어떻게" 세 요소 포함. 40자 이내
- **핵심 기능 1개**: 딱 하나. 두 번째를 제안하면 실격

**Step 3: 유저 플로우 (최대 4단계)**

돈을 내기까지의 흐름만 그린다. 부가 기능, 설정 화면, 온보딩 투어는 이 단계에 없다.

```
1단계: [랜딩/진입 — 어디서, 어떻게 오는가]
2단계: [핵심 가치 경험 — 돈 내기 전에 뭘 보는가]
3단계: [전환 트리거 — 무엇이 결제를 결심하게 만드는가]
4단계: [결제 — 어떻게 돈을 받는가]
```

4단계를 초과하면 하나를 제거한다. 제거할 수 없으면 핵심 기능 선택이 잘못된 것이다.

완료 후 보고: "Phase 1 완료: [제품명] 확정. 핵심 기능: [1개]. 유저 플로우 [N]단계."

산출물: `product.md`

### Phase 2: 설계 — 지형을 읽고 도구를 고른다

장인이 일을 잘 하려면 먼저 도구를 갈아야 한다. 도구 선택이 곧 일의 첫 번째 행위다. 잘못된 스택은 Week 1에 솔로 빌더를 무너뜨린다. 그리고 가격은 내 시간값이 아니라 구매자가 지금 쓰는 비용에서 나온다.

**Step 1: 기술 스택 선택**

선택 기준은 속도다 — 선호도, 트렌드, 확장성이 아니다. 두 가지 질문으로 판단한다:
- "배포까지 몇 시간이 걸리는가?"
- "14일 후에도 혼자 유지보수 가능한가?"

스택 제안 시 이유를 명시한다. 기존 인프라를 최대한 활용한다 — 처음부터 짓지 말고 거인의 어깨 위에 선다.

**Step 2: 핵심 기능 구현 로직**

Phase 1의 핵심 기능을 가장 단순한 경로로 구현하는 방법을 적는다.

AI를 사용하는 경우, 시스템 프롬프트 초안을 이 단계에서 함께 작성한다:
```
[시스템 프롬프트 초안]
역할: ...
제약: ...
출력 형식: ...
```

**Step 3: 가격 책정**

가격은 구매자가 이 문제를 해결하기 위해 현재 지출하는 비용에 비례해야 한다. 내 시간값 계산이 아니라 상대방의 고통 비용을 기준으로 잡는다.

세 가지를 순서대로 확정한다:
1. "구매자가 이 문제를 해결하려고 지금 쓰는 돈 또는 시간은?" — WebSearch로 기존 솔루션 가격 확인
2. 현재 지출의 몇 분의 1로 설정할지 결정 (통상 30~50%)
3. 무료로 줄 것 / 돈 받을 것 구분

```
- 무료: [신뢰를 먼저 쌓는 것]
- 유료: [___/월 또는 건당 ___]
- 목표: 출시 후 ___일 내 유료 ___명
```

완료 후 보고: "Phase 2 완료: 스택 [선택 스택]. 유료 가격 [금액]. 목표 [N일 내 M명]."

산출물: `design.md`

### Phase 3: 포고문 — 낯선 이를 움직이는 말을 찾는다

설득은 순차다: 신뢰 → 감정 → 논리. 신뢰가 없으면 감정이 작동하지 않고, 감정이 없으면 논리도 닫힌 귀에 닿지 않는다. 그리고 아무리 좋은 말도 타이밍이 틀리면 실패한다.

**Step 1: H1 (고통)**

읽는 사람이 "이거 나 얘기네"라고 느끼는 한 줄. 해결책을 말하지 않는다. 고통만 찌른다.

좋은 예시의 구조: [지금 겪고 있는 구체적 상황] + [그 고통의 핵심]
- 막연한 고통 ("시간이 없다") 대신 구체적 고통 ("미팅 녹취록을 정리하는 데 매일 2시간")

**Step 2: 서브헤드 (해결책)**

한 줄로 무엇이 해결되는지. Phase 1의 한 문장 설명을 그대로 쓰지 않는다 — 구매자 언어로 번역한다.

**Step 3: 신뢰 기반**

왜 내가 이걸 만들었는가 — 1~2문장. 자격증이나 경력이 없어도 된다. "나도 이 고통을 겪었다"가 종종 가장 강한 신뢰다.

**Step 4: CTA — 행동 하나**

구체적이고 낮은 진입 장벽. "시작하기" 대신 "지금 무료로 [핵심 가치 경험]해보기."

**Step 5: 타이밍 확인**

이 메시지가 가장 효과적으로 작동하는 맥락을 한 줄로 정의한다: "이 사람이 [어떤 상황]에 있을 때 이 페이지를 보면 전환된다."

타이밍을 알면 Week 2 유저 유입 채널을 선택할 때 기준이 생긴다.

완료 후 보고: "Phase 3 완료: H1 [헤드라인]. CTA [버튼 텍스트]. 타이밍 [맥락 한 줄]."

산출물: `landing.md`

### Phase 4: 14일 전쟁 — 첫 번째 드럼을 친다

전투는 기세의 문제다. 첫 번째 북소리에 기세가 솟구치고, 두 번째에 줄어들고, 세 번째에 다한다. 솔로 빌더의 14일도 같다. Week 1에 배포하지 않으면 Week 2는 약해진다. 속도가 유일한 자산이다.

**Week 1 — 핵심 기능 완성 + 랜딩 배포**

첫 번째 드럼 소리다. Day 7 이전에 실제 URL이 존재해야 한다.

1. Day 1–2: 스택 세팅 + 첫 화면 작동
2. Day 3–4: 핵심 기능 End-to-End 작동 (거칠어도 된다)
3. Day 5: 결제 플로우 연결 (실제 결제가 되어야 함)
4. Day 6: 랜딩 페이지 배포 + 기본 QA (데스크탑/모바일)
5. Day 7: 첫 번째 유저 10명 직접 초대 (광고 아님, 직접 접촉)

**Week 2 — 유저 유입 + 피드백 + 전환**

낯선 이를 고객으로 바꾸는 순서: 먼저 공감하고, 가치를 먼저 주고, 차별화를 드러내고, 마지막에 가격을 제시한다.

1. Day 8–9: 공감 — 타겟 커뮤니티에서 고통을 공감하는 글 작성. 광고가 아니라 "나도 이 문제 겪었다" 류의 글. Phase 3의 타이밍이 채널 선택 기준이 된다.
2. Day 10: 가치 먼저 — 직접 DM 또는 무료 체험 제공 (10~20명). 가치 먼저, 설명 나중.
3. Day 11–12: 차별화 — 첫 유저 피드백 수집. "왜 안 썼는가"가 "왜 좋은가"보다 중요하다. 메시지와 기능을 정제한다.
4. Day 13: 전환 — 유료 전환 제안. 직접 연락한 사람들에게 먼저. 가격은 Phase 2에서 확정한 그대로.
5. Day 14: 결과 집계. Phase 5 판정 준비.

완료 후 보고: "Phase 4 완료: Week 1 배포 [완료/지연]. Day 14 유료 전환 [N명]. Phase 5로 이동."

산출물: `sprint.md`

### Phase 5: 판정 — 지금 결정한다

전략적 후퇴가 최선인 조건은 사전에 확정해야 한다. 결과가 나온 뒤 판단하면 감정이 판단을 흐린다. 통제할 수 있는 것과 없는 것을 먼저 분리하라.

**Step 1: 성공 지표 — 숫자 하나만**

첫 달에 볼 숫자 딱 하나. 두 개를 쓰면 판단이 흐려진다.

```
성공 지표: ___ (예: Day 30까지 유료 전환 N명 / MRR $X)
```

**Step 2: 통제 가능/불가능 분류**

피드백이 들어왔을 때 이 질문 하나로 분류한다:

"이 피드백은 내가 바꿀 수 있는 것(메시지, 가격, 채널, 기능)에서 왔는가?"
- **그렇다** → 조정하고 계속한다.
- **아니다** (수요 자체 없음, 예산 없는 시장, 구조적 진입 장벽) → 전략적 후퇴를 고려한다.

**Step 3: 피봇 트리거 — 지금 확정한다 (AskUserQuestion)**

결과가 나온 뒤 결정하면 늦다. 지금 조건을 써놓는다.

"아래 피봇 트리거를 확인하고 수정할 부분이 있으면 알려주세요:"

```
피봇 트리거: "Day [N]까지 [성공 지표의 X%]를 달성하지 못하고,
[통제 불가능한 신호 하나 — 예: 10명에게 직접 제안했는데 0명 응답]이면
즉시 방향 전환한다."
```

피봇은 포기가 아니다. 다음 결전을 위한 전략적 이동이다.

완료 후 보고: "Phase 5 완료: 성공 지표 [지표]. 피봇 트리거 [조건]. 빌드 플랜 완성."

산출물: `verdict.md`

## Output Structure

```
~/.sblg/build/
└── {제품슬러그}-{YYYYMMDD}/
    ├── _status.md
    ├── product.md
    ├── design.md
    ├── landing.md
    ├── sprint.md
    └── verdict.md
```

## _status.md Format

```markdown
# Session Status
- product: {제품명}
- current_phase: {0~5}
- started_at: {날짜}
- last_updated: {날짜}
- core_feature: {핵심 기능 한 줄}
- stack: {선택 스택}
- price: {가격}
- success_metric: {성공 지표}
- pivot_trigger: {피봇 조건}
- week1_deployed: {true/false}
- paid_conversions: {N}
- next_action: {다음에 해야 할 구체적 작업}
```
